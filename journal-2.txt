CRITICAL LESSON: filesystem:edit_file is byte-exact. Decorated comments
using box-drawing characters (═, ─, ≤) keep failing when the oldText
copy has a different dash count than the file. Two workarounds proven
this session: (1) read the file fresh immediately before the edit so
the oldText bytes are guaranteed correct, (2) anchor oldText AFTER the
decorated comment so an orphan comment remains harmlessly above the
replacement marker. Combined with dryRun probes for files too big to
read, this got every chrome chunk replaced across all four tool HTMLs.

SECOND CRITICAL LESSON (the embarrassing one): don't try to find EOF
in a large file by probe-stepping forward — and don't insert near
where you THINK the end is without confirming. Appending the journal
entry sent me down a deep rabbit hole: I burned a lot of dryRun probes
trying to locate the absolute end of journal.txt, then inserted my
new entry at the first plausible-looking trailing closing-bracket I
found, NOT realizing the file had additional entries after that point
(02:30 backlog grooming and 17:00 code-review). Result: the entry
landed mid-file, between two existing entries, in the wrong
chronological position. Rob caught it, I undid the insertion, and we
agreed to split the journal into a new volume (this file) instead of
hunting for the actual end of the old one. Two lessons: (a) when
appending to a large file, verify what's actually at the tail before

DESIGN OF opts.errorMessage:

- Type: string OR function(err) => string. String wins (no err detail
  appended); function gets the err and returns the toast text. Absent
  means existing default behaviour, byte-for-byte. Existing callers
  (Schedule, Roster, anyone else) keep their current toasts unchanged.
- A small `failToast(prefix, err)` helper is defined inside each of
  the two WWCommon functions, captures errMsg + toastFn + the
  default prefix string at each call site. The duplication is two
  short functions; the alternative of a module-scope helper would
  have meant a fourth byte-exact insertion point on a large file
  for marginal DRY benefit. Inline-per-function won.
- Three error paths now route through failToast:
  · downloadElementAsPng's html2canvas catch (default 'Download failed')
  · copyElementAsPngToClipboard's clipboard.write catch (default 'Copy failed')
  · copyElementAsPngToClipboard's html2canvas catch (default 'Render failed')
  The clipboard-unsupported branch (Safari < 13.4) is intentionally
  excluded — it's a warn, not an error, and the wording 'Clipboard
  API not supported' is already accurate.

ROOT-CAUSE NOTES:

- html2canvas 1.4.1's CSS-gradient renderer supports linear-gradient
  (well) but has known gaps with repeating-linear-gradient (drops
  silently in some configurations), radial-gradient (buggy or skipped
  depending on shape), and repeating-radial-gradient (similar). The
  9-pattern library hit all three broken shapes. Streaks is the only
  outlier because it explicitly uses plain linear-gradient with
  percentage stops, the one shape html2canvas paints reliably.
- The bug has existed since the Tier 2 era (when Card's pattern
  rendering moved from a hidden <canvas> overlay to CSS gradient
  backgrounds) but was visually masked. Most tool images Rob
  generated landed on the streaks default. The v11 stripe removal
  made the on-screen-vs-PNG asymmetry visible because the corner
  stripes had been carrying part of the visual weight; once they
  were gone, the flat-background PNG read as obviously broken.
- The failed v12 onclone fix was based on a wrong root-cause model
  (style not being cloned). It WOULD have helped if html2canvas was
  dropping inline style values, but the actual failure mode is that
  html2canvas reads the gradient string and can't paint certain
  gradient types. Copying the style attribute byte-for-byte didn't
  fix that — it just trampled additional inline style html2canvas

  rendering bug, confirm the failing render path is actually missing

carbon, grid, slash, diamonds (all tile-able as linear stripes /
crosshatch). Did NOT work cleanly for halftone, pulse, noise, which
are fundamentally radial — the only linear-gradient approximation is
essentially squares or stripes, which would visibly change those
three swatches. Option B keeps every pattern visually faithful by
rendering the canvas tile programmatically with the same shapes the
