CRITICAL LESSON: filesystem:edit_file is byte-exact. Decorated comments
using box-drawing characters (═, ─, ≤) keep failing when the oldText
copy has a different dash count than the file. Two workarounds proven
this session: (1) read the file fresh immediately before the edit so
the oldText bytes are guaranteed correct, (2) anchor oldText AFTER the
decorated comment so an orphan comment remains harmlessly above the
replacement marker. Combined with dryRun probes for files too big to
read, this got every chrome chunk replaced across all four tool HTMLs.

SECOND CRITICAL LESSON (the embarrassing one): don't try to find EOF
in a large file by probe-stepping forward — and don't insert near
where you THINK the end is without confirming. Appending the journal
entry sent me down a deep rabbit hole: I burned a lot of dryRun probes
trying to locate the absolute end of journal.txt, then inserted my
new entry at the first plausible-looking trailing closing-bracket I
found, NOT realizing the file had additional entries after that point
(02:30 backlog grooming and 17:00 code-review). Result: the entry
landed mid-file, between two existing entries, in the wrong
chronological position. Rob caught it, I undid the insertion, and we
agreed to split the journal into a new volume (this file) instead of
hunting for the actual end of the old one. Two lessons: (a) when
appending to a large file, verify what's actually at the tail before
picking an anchor — don't assume any closing bracket is THE closing
bracket; (b) when a file is too big to read in one shot and probing
is taking many tool calls, that's a signal to split it, not to keep
probing. Should have suggested the volume-split immediately when the
read overflowed.

WHAT CHANGED IN common.js:

- downloadElementAsPng(node, opts) gained three opts. `filename` is a
  full filename override that wins over the auto-built
  `prefix + slug + date.png`. Card needs this because its filenames
  follow a fixed `ww_YYYYMMDD_scorecard.png` template generated by the
  tool, not the prefix+date pattern Schedule and Roster use. `before`
  is a sync callback fired after `body.classList.add('exporting')` but
  before html2canvas runs; Card uses it to hide the drag-tip overlay.
  `after` is a sync callback fired after html2canvas resolves (in both
  success and catch paths) so a tool can restore state symmetrically.
  Both hooks are try/catch-wrapped so a thrown error inside a hook
  doesn't take down the whole export.
- copyElementAsPngToClipboard(node, opts) got the same three opts via
  the exact same wiring, for symmetry.
- Public-API comment at the bottom of common.js updated from "Card-
  builder helpers (Schedule + Roster patterns; all three for download)"
  to "Card-builder helpers (shared by all three card tools)" since
  Card now uses the download path too.
- applyPatternToElement docstring updated to reflect that all three
  card tools now share it (was "Schedule and Roster").

WHAT CHANGED IN Card.html:

- CSS: .copy-btn.success renamed to .copy-btn.done. The class WWCommon
  adds during the "Copied!" flash is `.done`; Card was the only tool
  still using the historical name `.success`. Same color
  (background:#1a7a3a) so the visual is unchanged.
- JS: download() and copyToClipboard() rewritten as thin wrappers. The
  drag-tip hide moved out of the body of each function into a shared
  hideDragTip() helper that's passed as the `before` hook. The old
  setTimeout(50ms) wait was dropped — html2canvas reads computed
  styles synchronously, so the before-hook's display:none is visible
  without needing a reflow tick. activeCardEl() and downloadFilename()
  stayed put — those are Card-specific data lookups, not plumbing.
- The downloadInProgress re-entrancy guard stayed put on Card.
  WWCommon handles disabling+re-enabling the button on its own, but
  Card's guard is set across the await so a double-click during the
  Promise window can't fire two parallel downloads.
- Minor wording change accepted: WWCommon's toast() messages on failure
  ("Download failed: ..." / "Copy failed: ...") replace Card's old
  friendlier wording ("Could not generate image — try the Download
  button instead"). Worth it for shared error handling.
