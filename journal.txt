paw renders at fixed 71×92 px (naturally portrait, ~0.77 aspect, area
6,532 px²); the opponent logo goes through calcFit(natW, natH, 92), so a
square or near-square opp logo fills a full 92×92 (area 8,464 px², ~30%
more visual area — reads as bigger to the eye). Fix: derive the opp box

shrink proportionally too, so balance holds for any aspect ratio. Bonus:
because the constants are now derived from the paw dimensions rather

automatically. Final + Halftime cards share OPP_BOX_SCORE; Preview uses

absolutely-positioned #card-frame* border (sized to 100% × 100% of
whatever-was-captured) stopping at the clip edge, exactly matching the
visible browser viewport rather than the card's natural 440 px width.
Desktop unaffected because the overflow clip only exists on mobile.

Approaches considered before picking the fix: (1) CSS transform: scale()
on the wrapper to shrink the visible card into viewport width — rejected
because html2canvas reads real rendered pixel dimensions, so the PNG
would have come out at e.g. 374×374 instead of 440×440 (lower-resolution
export); (2) pass html2canvas the windowWidth: 440 option —
insufficient, in v1.4 windowWidth affects vw-unit calculations but
doesn't re-evaluate already-applied media queries against the simulated
window, so the mobile CSS state remains active during capture; (3)
temporarily reposition the wrapper to position:fixed off-screen,
capture, restore — works but more intrusive (touches positioning, breaks
the wrapper's layout context for absolute-positioned children, harder to
reason about across 3 files with different download paths); (4) one-shot
body.exporting class toggle — chosen as the most surgical option.

Implementation: a single CSS rule inside each file's mobile @media
block: body.exporting .card-container { overflow: visible !important;
width: auto !important; max-width: none !important; align-self:
flex-start !important; flex-shrink: 0 !important; padding-left: 0
!important; padding-right: 0 !important; } Plus 2 JS lines added around
each html2canvas() call: document.body.classList.add('exporting')
immediately before, document.body.classList.remove('exporting') as the
very first thing inside both the .then and the .catch (so the class is
gone the moment html2canvas resolves, before any blob/clipboard work
happens). Six call sites total: Card.html has 2 (download() +
copyToClipboard()), Schedule.html has 2 (downloadCard() +
copyToClipboard()), Roster.html has 2. Final add/remove counts per file:
2 adds + 4 removes (one remove per success path, one per catch path, ×2
functions). Validated: HTMLParser passes on all three, node --check
passes on all 9 inline script blocks. align-self:flex-start was needed
because .app-body's column flex defaults align-items:stretch, which
would otherwise stretch the now-unconstrained container back to body
width.

Why the !important flags: the mobile @media block sets .card-container
width:100%, overflow-x:auto, and the existing desktop CSS sets
justify-content/align-items in the same selector. Without !important,
specificity would tie and source-order would let the @media block win
since body.exporting .card-container has equal class+element specificity
as just .card-container plus an extra body.X class — actually that gives
body.exporting higher specificity (2 classes + 1 element vs 1 class), so
!important is technically defensive overkill, but it makes the override
intent unmistakable to future readers.
